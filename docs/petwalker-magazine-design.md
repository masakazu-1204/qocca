## 0. コンセプト — 「データベース」の上に「編集部」を載せる

現状のペットウォーカーは *検索する道具*（エリア→一覧→詳細）。
本設計は既存の道具を壊さず、その上に *読む楽しみ* の層を足す。

```
        ┌─────────────────────────────┐
        │  特集層 (新設)  = 雑誌の記事     │  ← 編集の意志がある。読む。
        │  ①エリア特集 ②テーマ特集 ③コース │
        └──────────┬──────────────────┘
                   │ スポットを「登場させる」
        ┌──────────┴──────────────────┐
        │  データ層 (既存)  = 1,551スポット │  ← 検索する。調べる。
        │  エリア一覧 / 近くのおでかけ先 / 詳細 │
        └─────────────────────────────┘
```

- 特集はスポットの**編集された見せ方**であり、スポットデータの複製ではない
  （記事の中のスポットは spot_id 参照。説明や条件は常に最新のDBを映す）
- 既存3層UI・GPS検索・お気に入りは**一切壊さない**（特集層は純増設）

## 1. 3タイプの構造定義

### ① エリア特集 (type: `area`) — 「高尾・奥多摩 — 犬の神様が待つ山へ」
その土地を1本の物語として読ませる。**雑誌の巻頭特集**にあたる。

```
[ヒーロー] 全幅写真 (既存エリア背景 or 専用) + 英字ラベル + タイトル + サブタイトル
[リード文] 300〜400字。土地の空気・なぜ犬と行くべきか (詩的・静けさトーン)
[章 × 3〜5] 見どころスポットの深掘り。写真 + 見出し + 本文200〜300字 + スポットカードへの接続
[コース内包] 「この山の歩き方」= モデルコース1本へのリンク (Phase3以降)
[すべての場所] 既存のエリア特集一覧 (カテゴリタブ付き) へ静かに接続
```

### ② テーマ特集 (type: `theme`) — 「犬の神様に会える神社」「海が見えるドッグラン」
エリアを横断してスポットを束ねる。**季節・切り口の企画記事**。

```
[ヒーロー] + [リード文 200〜300字]
[スポット列 × 5〜10] エリア横断。各スポット = 写真 + 一言 (なぜこの特集に入るか) + 詳細へ
[季節性] season_tags ('spring'|'summer'|'autumn'|'winter'|'all') で棚の出し分け
```
※ 既存の「目的で探す」(高速SA/道の駅/雨の日OK) は *恒常テーマの原型*。
   テーマ特集はその発展形 — タグの束ではなく「編集の一言」がつく。

### ③ モデルコース (type: `course`) — 「愛犬と巡る、みなとみらい1日プラン」
時系列 + 実用。**巻末の実用ページ**にあたる。

```
[ヒーロー] + [リード文 150〜200字 (このコースの気分)]
[ステップ × 4〜7] time_label ("10:00") + スポット + 一言 (移動・注意・ごはん)
   ※ 昼にペット可レストラン (category:cafe) を必ず1つ挟む構成ルール
[コース地図] 全ステップのピンを PetWalkerMapView (Phase3実装済) 流用で1枚に
[実用フッター] 所要時間目安・雨の日代替 (theme特集へのリンク)
```

### 3タイプの連動（雑誌の階層）
```
トップ「特集」棚 ─→ ①エリア特集 ─内包─→ ③モデルコース
                 └→ ②テーマ特集 ─→ 各スポット詳細 ─逆リンク─→ 「この場所が登場する特集」
```
- エリア特集がハブ。テーマ特集は横糸。コースは実用の出口
- スポット詳細に「この場所が登場する特集」を将来表示（Phase4・逆引き）

## 2. データ活用設計

### 新テーブル2つ（King承認必要 = スキーマ追加）

```sql
-- 特集記事 (3タイプ共通)
pw_features (
  id uuid PK,
  slug text UNIQUE,               -- 'takao-oinusama' (URL/共有用)
  type text CHECK ('area'|'theme'|'course'),
  title text,                     -- 「高尾・奥多摩 — 犬の神様が待つ山へ」
  subtitle text,                  -- 英字ラベル "The Mountain of the Dog God"
  lead_text text,                 -- リード文
  hero_image_url text,
  area_tag text,                  -- area/course型のみ。既存 PW_AREAS.tag と一致
  season_tags text[] DEFAULT '{all}',
  status text CHECK ('draft'|'published') DEFAULT 'draft',
  sort_order int, published_at timestamptz, created_at timestamptz DEFAULT now()
)

-- 記事の章 / コースのステップ (1テーブルで両対応)
pw_feature_sections (
  id uuid PK,
  feature_id uuid,                -- pw_features へ (アプリ側担保 or FK)
  sort_order int,
  heading text,                   -- 章見出し「おいぬ様に、会いに行く」
  body_text text,                 -- 本文 200〜300字
  spot_id uuid,                   -- pet_walker_spots へ (nullable: スポットに紐づかない章もOK)
  image_url text,                 -- 省略時は spot の image_urls[0] を表示 (常に最新画像)
  time_label text                 -- course型のみ "10:00"。area/themeはNULL
)
```

- **2テーブルで3タイプ全部**表現できる（course = sections に time_label / theme = sections が横断スポット列）
- RLS: published のみ公開SELECT・書込は service_role のみ（管理UIは作らない・当面SQLで投入）
- spot_id 参照により、スポットの説明/条件/画像の更新が**記事に自動反映**される

### 記事は手動か半自動か → **「半自動生成 + King の目」を推奨**
| 方式 | 評価 |
|---|---|
| 完全手動 | 品質最高だが King の時間が消える。60エリア×3タイプは回らない |
| 完全自動 | 世界観が死ぬ。静けさの文章はテンプレでは書けない。**却下** |
| **半自動 (推奨)** | クマが description/source_note を素材にリード文・章文を執筆 → **draft投入 → King確認 → published**。ブログの生成運用と同じ信頼モデル |

素材が既に詩的トーン（「苔むす岩と清流に沿うハイキング道」等）で書かれているため、
記事文はその延長線で書ける。**1本 = クマ執筆30分 + King確認5分** の想定。

## 3. UI設計（静けさ世界観・エディトリアル）

### petwalker.tsx との関係 — 既存3層は不変、記事viewを純増設
```
既存: エリア一覧 → エリア特集(タブ一覧) → スポット詳細   ← 全て不変
追加: トップに「特集」棚 + 新view `feature` (PW_NAV pushState パターン踏襲)
```
- トップ構成（上から）: ヒーロー → **特集棚(新)** → 保存した場所 → 近くのおでかけ先 → エリアで探す → 目的で探す
- 特集棚 = 雑誌の表紙が並ぶ棚。**非対称レイアウト**（1本目を大きく、2本目以降は小さく横並び）

### エディトリアル様式（QC様式の範囲内でやる）
- 見出し: QC_FONT_DISPLAY (Shippori Mincho系) を**大きく・weight 500以下**
- 英字サブタイトル: QC_FONT_EN・letter-spacing 3〜4 (既存作法)
- 本文: QC_FONT_JP・fontWeight 300・lineHeight 2.2・**1段組・最大幅 640px**（読み物の幅）
- 写真: 章ごとに全幅⇄半幅を交互に（非対称のリズム）。テキスト回り込みはしない
- 余白: セクション間 120〜200px。**「白で読ませる」**
- 禁止事項の再確認: 絵文字なし・#F5A94A不使用・純黒/純白なし・transition 0.8s+・煽り表現なし
- モデルコースの時刻ラベルは装飾でなく **QC.sage の小さな英字**（"10:00 AM" 風の静かな表記）

### トップ「街の読みもの」(blog_posts) との棲み分け
| | 街の読みもの (blog) | PW特集 (新設) |
|---|---|---|
| 主題 | **人と暮らし**（豆知識・健康・クリエイターの裏側） | **場所と旅**（エリア・テーマ・コース） |
| 構造 | author基点・単一本文・SNS連携 | スポットDB結合・章構造・spot_id参照 |
| 場所 | ホーム/ブログページ | ペットウォーカー内 |
| 例え | 街の回覧板・コラム | 旅の雑誌 |
- テーブルも画面も**完全分離**（blog_postsへの相乗りはしない: author/SNS列が場所記事に不適合）
- 相互リンクは可（ブログ記事「高尾に行ってきた」→ 高尾特集、など将来）

## 4. GPS現在地検索との連携（Phase4・オプション）

素材は揃っている（スポット座標99.6% + PetWalkerMapView実装済）:
1. **コース地図**: モデルコースのステップを地図に描く（Phase3で使用・流用のみ）
2. **近くの特集**: nearby view の末尾に「この近くの特集」カードを1枚
   （特集の登場スポット群の重心と現在地の距離で判定・client計算で足りる）
3. **特集内の距離表示**: 特集を現在地つきで開いたら各章に「約3.2km」を薄く添える
- いずれも**追加であって前提ではない**。Phase1-3はGPSなしで完結する
