# ペットウォーカー 188エリア計画 設計書 v1

> 2026/7/6 作成。棚卸し(現状643件・44タグ・37県カバー)にもとづく 643→3,000件 拡充ロードマップ。
> **King確定方針**: 二層命名(UI=観光地名 / 管理=県×方位裏タグ)・既存タグ温存・優先順=首都圏→深掘り→空白県・超広域(九州/北海道)は分割。

## 0. 設計原則

1. **旅心優先**: ユーザーが見るのは常に観光地名タグ(「淡路島」)。「兵庫南」は見せない
2. **裏タグ**: `secondary_area_tags` に県×方位コード(日本語「兵庫南」形式)を持たせ、網羅管理・集計に使う
3. **実態主義**: 188マス全埋めはしない。観光地/動物関心が薄いマスは「なしでOK」と宣言して空けておく
4. **1マス複数タグOK**: マスは管理単位。神奈川南に「鎌倉」「湘南」が同居してよい
5. **テーマタグ**(高速SA/道の駅/雨の日)は188マスの外の別系統(kind:"theme")

## 1. Step1: 188マス定義

- 47都道府県 × 東西南北 = **188マス**
- 識別子: `県名(都府県なし)+方位1字` — 例: `兵庫南` `東京西` `北海道東`
- 方位の割り方: 県庁所在地や中央部の地物は**最寄り方位に寄せる**(「中央」マスは作らない)。例: 仙台→宮城南、金沢→石川南
- 県をまたぐ観光圏(蔵王・しまなみ等)は**主マス1つ+副マス**(裏タグを複数持たせる)

## 5. Step5: 標準フロー運用設計(1マス=1便)

```
① ワイ (claude.ai): Web調査・一次ソースで 10-15件リストアップ (名称/カテゴリ/住所/ペット可否根拠)

   - constants/petwalker.ts に PWArea 追加 (tag=観光地名・slug・en・blurb)

     (area_tag=観光地名 / secondary_area_tags=['県方位'] / pref/city/緯度経度/approval_status='approved')
   - 検証SELECT (件数・タグ一致・pref分布)
③ ちゃぴ (ChatGPT): 背景画像生成 → King が WebP変換 (出力先: petwalker-out/)
④ King: Supabase Storage petwalker/<slug>.webp にアップ → UI自動反映 (URL決定的)
```

- チェックリスト(毎便): DB↔constants タグ完全一致 / slug重複なし / pref表記は「〇〇県」統一 / 裏タグ付与 / 実在・営業中・ペット可の一次ソース確認
